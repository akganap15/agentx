## What changed this session

### 1. Backend logging

- Added `logging.basicConfig(...)` to `backend/server.py` so application
  loggers actually emit to the console. Previously only uvicorn's own logger
  was wired up, which is why `logger.info(...)` calls were silently
  discarded. Format includes timestamp + level + module name.
- Silenced noisy third-party loggers: `httpx` and `httpcore` set to WARNING
  (otherwise every OpenAI request shows up as INFO).

### 2. Retell transcript logging — real-time

In `backend/src/api/routes/retell_voice.py`:

- Tracks `latest_transcript` and `logged_turn_count` per WebSocket session.
- When a new turn appears in the cumulative transcript Retell sends on each
  message, the *previous* (now-finalized) turn is logged via
  `_log_turn(entry)` as:
  `Retell turn: call_id=... [agent|user] <content>`
- The final in-progress turn is flushed in the `finally` block when the
  socket disconnects.
- All other Retell-related INFO logs were removed per user request:
  no more `Retell WS message`, `Retell reply`, `Retell WebSocket connected/
  disconnected`, `Retell call ended`, `Retell closed socket before reply
  could be sent`. Error/exception logs are still kept.
- An earlier attempt routed the transcript through a frontend POST to a new
  `/api/v1/voice/retell/log-transcript` endpoint (added to the same file
  along with corresponding code in `frontend/tablet/src/VoiceCallRetell.jsx`).
  The endpoint and frontend code still exist but are now redundant — the
  backend WS handler logs everything we need directly. **Safe to remove if
  you want to clean up.**

### 5. Hard-coded demo appointment for the dashboard

Created `backend/src/db/seed_appointment.py`. On every server startup it:

1. Inserts a `Conversation` into the store with
   `outcome="appointment_booked"` so the dashboard "Appointments Booked"
   KPI (`backend/src/api/routes/dashboard.py:93`) shows ≥ 1 immediately.
2. Calls Google Calendar `events.insert` directly (uses an
   `AuthorizedSession` from the same service-account credentials as
   `backend/src/tools/calendar.py`) to create a real event:
   - Customer: **Kelly Ann Smith** (was Sarah Mitchell first, renamed
     mid-session)
   - Phone: `+15555550123`
   - Service: **Base Color, Highlights, Haircut & Blowout**
   - Time: **6:00 PM tomorrow**, 3 hours long, `America/Los_Angeles`
   - Location: Ashwin's Hair Studio
   - Calendar: whatever `GOOGLE_CALENDAR_ID` is set to in `.env`
     (currently `ashwinnnishad@gmail.com`)
   - Event ID: deterministic per day — `demoappt<YYYYMMDD>` — chars are
     base32hex-safe (a–v + 0–9). Restarting the server multiple times the
     same day hits a 409 and skips re-creating.
3. Errors from the Calendar API are caught and logged as warnings —
   startup never fails because of a calendar issue.

The seed is wired into `server.py` lifespan in **both** the in-memory and
SQLite branches.

**Caveat**: because the calendar event ID is deterministic per-day, renaming
"Sarah Mitchell" → "Kelly Ann Smith" mid-day did *not* update today's
existing Google Calendar event. To get the new name on today's event you
have to delete it manually first (or restart on a new day).
